# 通信対戦化メモ

## 結論

スマホ通信対戦は可能です。
ただし、押し合い物理ゲームは同期ズレが出やすいので、最初は「サーバー authoritative」方式にします。

## 推奨構成

```text
スマホA / スマホB / スマホC / スマホD
  ↓ 入力だけ送信
WebSocketサーバー
  ↓ 物理・衝突・落下・勝敗を計算
全スマホへ gameState を配信
```

## なぜサーバー側で計算するか

各スマホで物理演算すると、端末差や通信遅延で位置がズレます。
「自分の画面では落としたのに、相手画面では生きている」が起きるため、勝敗だけはサーバーで決めます。

## フェーズ案

### v0.1

スマホ1台ローカル2人対戦。
現在の版。

### v0.2

ゲーム本体を以下に分離。

- `core.js`: 物理、勝敗、ステージ判定
- `render.js`: Canvas描画
- `input.js`: タッチ/キーボード入力

### v0.3

同じWi-Fiやローカルサーバーで2人通信。
Node.js + Socket.IO で検証。

### v0.4

Cloudflare Workers + Durable Objects でルームコード対応。

### v0.5

4人通信、切断復帰、観戦、部屋一覧なしのコード入室。

## 通信時に送るデータ

### クライアント → サーバー

```json
{
  "type": "input",
  "roomId": "ABCD",
  "playerId": "p1",
  "seq": 120,
  "x": 0.5,
  "y": -0.2
}
```

### サーバー → クライアント

```json
{
  "type": "state",
  "tick": 450,
  "timer": 42.3,
  "players": [
    { "id": "p1", "x": 365, "y": 500, "vx": 20, "vy": 0, "alive": true },
    { "id": "p2", "x": 635, "y": 500, "vx": -20, "vy": 0, "alive": true }
  ]
}
```

## 注意点

- スマホ通信では完全リアルタイムより「多少ラグっても勝敗がズレない」を優先する
- プレイヤー側は受け取った位置へなめらかに補間する
- 部屋コードは4〜6文字で十分
- 途中切断したらCPU化か即負けのどちらかを決める
